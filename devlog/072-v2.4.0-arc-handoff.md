## 1. R1 / R2 / R3 patch arc — what each round addressed

The v2.4.0 Codex audit loop went three rounds. This was the longest audit arc on the project to date; the specific lock-file primitive was the hinge, and each round narrowed the invariant surface before the next round exposed the next layer.

### R1 — atomic `openSync('wx')` in `acquireInstanceLock`

- **What Codex flagged:** `acquireInstanceLock` used check-then-write (`existsSync` → `readFileSync` → `kill(0)` → `writeFileSync`). Two daemon processes starting concurrently both saw "no lock file" + both wrote.
- **What I shipped:** `openSync(pidFile, 'wx', 0o600)` for atomic exclusive-create at the syscall level. On `EEXIST`: read PID + `kill(0)`, liveness-probe → reclaim stale (`unlink` + retry ONCE, bounded by an `attempts>1` guard).
- **What also landed in the same round (unrelated):** HIGH #2 split-brain config (`loadConfig` was using flat path while `RELAY_DB_PATH` was per-instance — fixed via new `resolveInstanceConfigPath()` mirroring the DB-path resolver) + MED prompt-parameter injection (added per-argument `validate` regex on `McpPromptArgument` + `JSON.stringify`-escape the free-text `brief` field).

### R2 — fail-closed on every EEXIST (SECURITY hardening)

- **What Codex flagged:** the R1 unlink-on-stale path introduced a NEW TOCTOU race:
  1. Initial `instance.pid` = 999999 (stale).
  2. Process A: `wx` EEXIST → probe ESRCH → **pauses** before `unlink`.
  3. Process B: same path → unlinks → `wx` wins → writes its **live** PID.
  4. Process A resumes → unlinks B's LIVE pidfile → `wx` wins → writes its own PID.
  5. Both believe they hold the lock. Invariant violated.
- **What I shipped:** removed the auto-reclaim path entirely. Every `EEXIST` now fails-closed. Live holder → `"already running (PID N)"`. Dead/cross-user/unreadable → error with a verbatim `rm <path>` command for manual cleanup. No unlink ever runs inside the lock function.
- **Rationale — victra's call, which I agreed with:** "Fail-closed is the conservative cornerstone. Auto-reclaim can return as v2.5+ with the proper primitive AND a regression covering the exact 'stale reader pauses, other process acquires, stale reader resumes' schedule Codex demonstrated. Don't ship a half-fixed reclaim." Aligns with `memory/feedback_security_cornerstone.md`.

### R3 — POSIX-safe printed `rm` + CHANGELOG rewrite

- **MED — shell-injectable `rm`.** The R2 error text printed `rm "${pidFile}"` with double-quoted interpolation. Double quotes handle spaces but do **not** neutralize `$()`, backticks, or `$VAR`. Codex repro: a hostile `RELAY_HOME` path containing `$(touch SHOULD_NOT_RUN)` produced a copy-pastable command that, when pasted into a shell, executed the command substitution. Operator-controlled input so MED not HIGH, but it directly failed the "shell-safe printed rm command" ask from the R2 dispatch.
- **What I shipped:** new `shellSingleQuoteEscape(value)` helper in `src/instance.ts`. Canonical POSIX idiom: wrap in single quotes, escape interior `'` as `'\''`. Error + `log.warn` now emit `rm -- '<escaped>'`. Helper exported for test pinning.
- **LOW — CHANGELOG drift.** The HIGH #1 top bullet still described R1 behavior ("reclaim stale files (one retry bounded)") even though the R2 section correctly explained why auto-reclaim was removed. Rewrote the top bullet to match R2+R3 final shape.

### Why fail-closed (path A) over auto-reclaim-with-proper-primitive (path B)

Victra's dispatch explicitly offered both options in R2:
- **(A)** Fail-closed with clear manual cleanup guidance. "Slowest UX, fastest to ship, provably safe."
- **(B)** Reclaim primitive that cannot remove a different file/holder than just inspected (open-with-create-and-truncate guarded by lock-dir mkdir, or fcntl/flock on the pidfile fd). "More work, preserves auto-reclaim."

I picked (A). Three reasons, in priority order:

1. **Provability.** (A) is safe by construction — the lock function never calls `unlink`. I can point Codex at the function body and say "no mutation of an existing file; no race possible." (B) requires arguing about a specific primitive's semantics on macOS + Linux + Windows + NFS + whatever SMB adaptation, with a Codex-style schedule regression. That's a v2.5 design round, not a patch-round fix.
2. **Time budget.** R2 dispatch estimated 30–45min. Path (A) clocked in at ~25min. Path (B) would have needed ~3–5h of primitive research + cross-platform testing + a regression that actually exercises the interleave Codex demonstrated (which means forking or worker_threads + a pauseable mock in the middle of the reclaim).
3. **Operator-surface cost.** The stale-pidfile recovery UX is "verify the PID is dead, run the printed `rm`, retry." Not elegant but not operationally expensive — this happens after a crash + restart of a specific instance, which is rare. Operators who hit it once learn the routine.

The v2.5+ work for path (B) is captured in the punchlist below.

## 2. In-session debugging insights NOT in CHANGELOG

### R3 H1.3b — the double-shell-parsing trap

**The bug in my first draft:** I wrote the `shellSingleQuoteEscape` round-trip test using `execSync` with a JS template string:

```ts
const escaped = shellSingleQuoteEscape('$(touch X)');
const out = execSync(`sh -c "printf '%s' ${escaped}"`, ...);
expect(out).toBe('$(touch X)');
```

Assertion failed with `expected '' to be '$(touch X)'`. The file `X` got created. Why?

`execSync` passes the command string through a shell. Node's default `execSync` invokes `/bin/sh -c <cmd>`. That means my template string is evaluated TWICE: once by the outer shell that `execSync` spawns, once by the inner `sh -c` I wrote. In the outer shell, `$(touch X)` inside double quotes **expanded** before `sh -c` ever saw the escape. So the inner shell received `printf '%s' ''` — literally an empty single-quoted string — and `touch X` had already run as a side effect of the outer shell's expansion.

**The fix** was switching to `spawnSync` argv form:

```ts
const r = spawnSync('/bin/sh', ['-c', `printf %s ${escaped}`], { stdio: 'pipe' });
expect(r.stdout.toString('utf-8')).toBe('$(touch X)');
```

`spawnSync` with an argv array does NOT go through an intermediate shell. The argv items reach `sh` exactly as written; only the inner `sh -c` parses them. The escape is the thing being tested, not contaminated by an outer shell's expansion.

**Meta-lesson:** any test that asserts "a POSIX-escape round-trips through sh" MUST use `spawnSync` + argv array. If you see `execSync` with a JS template literal in a shell-escape test, it's probably wrong.

### R3 hostile-path regression — picking realistic path metacharacters

I seeded the hostile `RELAY_HOME` with `weird-$(touch SHOULD_NOT_RUN)-\`touch ALSO_NOT\`-with-'-quote-$USER`. Covers:
- Command substitution: `$(…)`
- Backtick substitution: `` `…` ``
- Variable expansion: `$USER`
- Single quote (forces the `'\''` escape path)
- Spaces (handled by single-quote wrap)

I explicitly left newlines out because `mkdir` on macOS rejects filenames with embedded `\n`. Single quotes around them would neutralize the newline but the test fixture couldn't create the path in the first place. If the punchlist ever adds the fcntl path, the regression should still include this set.

### Per-instance test isolation — `RELAY_HOME` env override vs `HOME`

First attempt at the per-instance tests tried to redirect `process.env.HOME = tmpDir`. Failed because `os.homedir()` doesn't always re-read `HOME` at call time — it can return the passwd-entry value even if `HOME` was changed mid-process. Switched to a dedicated `RELAY_HOME` env var honored by both `src/instance.ts botRelayRoot()` and `src/cli/init.ts defaultBotRelayDir()`. Production operators leave it unset and get `~/.bot-relay/`; tests set it and get an isolated sandbox.

This also shaped the lesson applied in R3: any test that needs to relocate `~/.bot-relay` uses `RELAY_HOME`; never override `HOME`.

### `fs.existsSync` vs `fs.lstatSync` for dangling-symlink active-instance pointer

`setActiveInstance(id)` creates a symlink whose target is the bare instance_id string (e.g. `work`), not an absolute path. On macOS, `fs.existsSync(linkPath)` **follows** the symlink and returns false when the target doesn't exist as a filesystem entity. So `isMultiInstanceMode` missed the active-instance pointer in the test.

Switched to `fs.lstatSync(linkPath)` which doesn't follow. Used this pattern everywhere we probe for the pointer's existence.

### The "backtick in a comment inside a template literal" trap (recurring)

Not v2.4.0 specific, but bit me again. `src/dashboard.ts` wraps all the client JS in a single JavaScript template literal. Any backtick in a comment inside that block — including in documentation or variable names — closes the template. Under v2.4.0 I didn't add code to `dashboard.ts` so I didn't re-hit this, but the pattern repeats across releases (v2.2.2, v2.2.3, probably earlier). Next build agent: if you're writing to `dashboard.ts` and typecheck reports a `',' expected` error on a comment line, the culprit is almost certainly a stray backtick in that comment.

## 3. Rejected alternatives

### POSIX double-quote + explicit dollar-escape (vs single-quote wrap)

**Idea:** instead of single quotes, double-quote the value and escape `$`, backticks, and backslashes inside.

**Why rejected:** double-quoted strings in POSIX sh still perform `$VAR` expansion even with `\$` escaping (which becomes `$` literal — correct) and command substitution requires escaping `$` + `` ` `` + `\` consistently across all versions of sh, dash, ksh, bash. Proving correctness is non-trivial. Single-quote wrap is the canonical idiom every Unix-person recognizes; reviewers can verify by inspection without running an audit matrix.

**Bonus:** single-quote wrap handles newlines and every other metacharacter automatically. Double-quote requires extending the escape set over time as new shell features appear.

### Atomic `mkdir`-lock (vs `fs.openSync('wx')`)

**Idea:** `fs.mkdirSync(lockDir)` succeeds-or-fails-atomically. Use a directory as the lock instead of a file; the directory name encodes the PID.

**Why rejected:** changes the on-disk shape from "the pidfile IS the lock" to "a directory exists when a daemon is active." Breaks backward compatibility with any operator tooling that reads `instance.pid`. Also complicates release semantics — `rmdir` has the same TOCTOU issues as `unlink` if you want auto-reclaim.

`fs.openSync('wx')` is the minimal primitive that delivers the "atomic create-or-fail" property while preserving the existing pidfile shape. Picked it.

### fcntl lock on the open fd (deferred to v2.5+)

**Idea:** after creating or opening the pidfile, acquire an advisory/exclusive fcntl lock on the open fd. The kernel releases the lock on process death, so stale locks self-clear without any unlink.

**Why rejected for v2.4.0:** node's standard `fs` module doesn't expose `fcntl(F_SETLK)`. You need either a native addon (e.g. `proper-lockfile`, `node-fcntl`) or a direct ffi binding. Adding a native dep to the core package is a distribution-model change. v2.5+ material with a proper RFC.

**v2.5+ note (see punchlist):** `proper-lockfile` is widely used and cross-platform via fallback to `mkdir`. Worth evaluating.

### Unlink + `fstat` inode check (TOCTOU mitigation without a proper primitive)

**Idea:** before `unlink`, probe the current inode (`fstat` on a freshly-opened fd). Only unlink if the inode matches what we saw during the liveness probe.

**Why rejected:** still racy. Another process could unlink + recreate the file between our `fstat` and our `unlink`, and the inode numbers could coincide (SQLite actually does this pattern with WAL files — new files can inherit recently-freed inodes). Plus inode numbers aren't guaranteed stable across all filesystems (NFS in particular). Not an improvement over fail-closed.

## 4. v2.5+ punchlist

These are real tasks waiting for future build cycles. None of them block the v2.4.0 ship; each has a specific reason it was deferred.

### 1. Auto-reclaim with proper atomic primitive

- **Scope:** restore "dead-holder stale-pidfile is automatically reclaimed" UX without re-introducing the R2 TOCTOU.
- **Primitives worth evaluating:**
  1. `proper-lockfile` npm package (cross-platform; falls back to `mkdir` on platforms without `fcntl`).
  2. A directory-based lock where the directory name encodes the pidfile's creation ctime or a uuid — unlink only the directory whose name matches what we just observed.
  3. `fcntl(F_SETLK)` via a native addon — kernel self-releases on process death, which is the only way to guarantee "a dead holder's lock is definitively dead."
- **Regression MUST mirror Codex's exact R2 schedule:** two processes observe the same stale pidfile; one pauses before its reclaim step; the other races through; the first resumes. Expected outcome: **AT MOST ONE** process ends up holding the lock. Easiest way to simulate: `child_process.fork` + a synchronization file that gates the pause point.
- **Ship target:** v2.5.x (not immediately — waiting on operator signal that the manual-cleanup UX is actually friction).

### 2. `http_secrets_previous` redaction gap in `src/transport/traffic-recorder.ts`

- **What Codex flagged in v2.4.0 audit:** the `REDACT_KEYS` allowlist misses `http_secrets_previous` (the rotation-grace list). Pre-existing — NOT new in v2.4.0 — but cheap to fix.
- **Fix:** add `'http_secrets_previous'` to the `REDACT_KEYS` set in `traffic-recorder.ts`. Add a regression asserting the field is `<REDACTED>` after capture.
- **Ship target:** v2.5.0 hygiene round. Can bundle with unrelated small items.

### 3. Markdown-backticks cosmetic (Codex R4 non-blocking)

- **What Codex noted in R4:** the error message wraps the rm command in markdown backticks (`` ` ``) inside a sentence that also contains prose. Some rendering contexts (terminals with auto-link detection, markdown-aware loggers) might strip the backticks or treat them as code-fence markers, which could subtly corrupt the printed command.
- **Fix shape:** either drop the backticks entirely in favor of a `\n\n    rm -- '<escaped>'\n\n` block format, or be explicit: `Run this command: rm -- '<escaped>' — verbatim.`
- **Ship target:** v2.5.0 polish. Cosmetic; no security impact.

### 4. Recurring v2.2.1 non-TTY daemon guard regression

- **Context:** the user's session-close message mentions "separate v2.4.2 brief covers the fix" for a TTY-guard regression that surfaced in live use after v2.4.0 ship. The v2.2.1 B3 fix + the regression-from-released-bugs suite were supposed to prevent this from re-landing.
- **Lesson (NOT a punchlist item per se, but a pattern to watch):** a regression file exists (`tests/v2-2-2-regression-from-released-bugs.test.ts`) and it HAS a case (2) for the non-TTY guard. Either that case was too loose (only checks source code string for `RELAY_SKIP_TTY_CHECK` + `process.exit(3)` via grep, not actually a runtime assertion), or the grep-based test missed a refactor that moved the guard to a different file.
- **Recommended v2.4.2 fix direction:** upgrade the regression-from-released-bugs case (2) from a source-code grep to a real subprocess run. Spawn `node dist/index.js` in a non-TTY context (vitest can do this with `child_process.spawn({stdio: ['ignore', 'pipe', 'pipe']})`) and assert exit code 3 + the expected stderr message. Plus a second case that spawns with `RELAY_SKIP_TTY_CHECK=1` and asserts the guard is bypassed.
- **Meta-lesson:** the regression-from-released-bugs pattern is only as strong as the assertions inside each case. Grep-based "structural" assertions catch file-removal regressions but not behavior drift.

## 6. Spawn handoff context — what the next build agent needs to know

Beyond what's in `CLAUDE.md`, `HANDOFF.md`, and `CHANGELOG.md`, here are session-specific observations worth carrying forward:

### The audit loop cadence is now "expect 2-3 rounds for hard invariants"

v2.2.1 → v2.2.2 was one round. v2.3.0 → v2.3.0-patches was one round. **v2.4.0 needed THREE rounds** because the lock-file invariant was genuinely subtle (R1 introduced R2's TOCTOU; R2 introduced R3's injection surface). Next time a patch is about a concurrent-access primitive, plan for multiple rounds and don't be surprised.

### Codex re-audit arrives as a fresh patch dispatch, not a thread reply

Each re-audit is a new `victra → victra-build` message with the full context inline. You don't need to re-read prior dispatches — the fresh one restates everything. Build agents who try to "catch up by reading old inbox" will duplicate work; the new dispatch is canonical.

### Fail-closed > speculative-auto-recovery when the primitive isn't proven

Strong default for security-sensitive paths. Operator UX takes a hit (manual cleanup instead of automatic), but the cornerstone of `memory/feedback_security_cornerstone.md` is that correctness-provable-by-construction beats "probably-correct under most schedules." If in doubt, fail-closed + queue the better-primitive work for a future release.

### `spawnSync` argv form is the default for shell-escape tests

Any test that verifies a string round-trips through `/bin/sh` MUST use `spawnSync('/bin/sh', ['-c', cmd], ...)` to avoid the JS-level shell expansion. `execSync` with a JS template literal is a trap (see §2 above). Carry this rule forward.

### `RELAY_HOME` is the canonical test-sandbox env override

Never override `HOME` in tests. `RELAY_HOME` is honored by `src/instance.ts botRelayRoot()` and `src/cli/init.ts defaultBotRelayDir()`; it points the whole `~/.bot-relay/` namespace at a tmp dir. Production operators leave it unset.

### Version budgets keep coming in under estimate

v2.3.0 estimated 24-32h, came in at ~8h. v2.4.0 estimated 16-20h, came in at ~6h. v2.4.0 patch arc (R1+R2+R3) estimated 1.5h + 30-45min + 15-25min = ~3h total, came in at ~80min. **Pattern:** the federation-design-memo's primitives (mailbox model, instance_id as UUID, MCP prompts/resources split) are all "obvious-once-designed" + fast to implement. The design memo IS doing the hard work. Trust the estimates' upper bounds as the hard-stop cap; the lower bounds are typically the realistic time.

### When Codex finds a drift in CHANGELOG or docs vs. code, FIX IN SAME PATCH ROUND

R3 bundled a MED (shell injection) with a LOW (CHANGELOG stale top-bullet) in one patch push. Per `memory/feedback_prefer_granular_tasks`, mixed-scope patches are fine for the same release arc. Don't defer cosmetic/doc drift to a separate round — it bloats the audit queue.

### Hall of Fame should credit the audit loop itself

v2.4.0 CHANGELOG credits Maxime + Codex + the v2.3.0 Part A infra. Future releases should also explicitly credit the re-audit loop — it's the mechanism that catches R1-induced R2 races. Without it, we'd have shipped broken concurrency code.
